## Audit: end-to-end signup ‚Üí bank ‚Üí bi-weekly reports

I traced every hop. The good news: the auth experience is fully de-Lovable-branded. The bad news: there are two real bugs that silently break the bi-weekly automation, plus a couple of cleanup items. Here's the honest state.

### 1. Sign-in (working, no Lovable branding)

- `/portal/login` is the only auth page. Two paths:
  - **Google** via `lovable.auth.signInWithOAuth("google", ‚Ä¶)` ‚Äî uses Lovable Cloud's managed Google OAuth (consent screen shows *your* app, not "Lovable"). This is the recommended provider; nothing user-facing says Lovable.
  - **Email + 6-digit code** via `send-login-otp` ‚Üí `verify-login-otp`. Code is generated server-side, SHA-256 hashed in `login_otps`, emailed through Resend from `Goldfin Desk <noreply@goldfindesk.com>`. On verify, server mints a magic-link `token_hash` and the browser redeems it for a Supabase session.
- All legacy paths (`/portal/signup`, `/portal/forgot-password`, `/portal/reset-password`) collapse into the same login page. No password UI anywhere.
- Verdict: signup never goes through any Lovable-branded screen. ‚úÖ

### 2. Plaid bank connection (working)

- `Accounts.tsx` ‚Üí `PlaidLinkButton` ‚Üí `plaid-create-link-token` ‚Üí Plaid Link ‚Üí `plaid-exchange-public-token` writes `plaid_items` + `plaid_accounts`. OAuth bank redirects handled. Env switches sandbox‚Üîproduction via `PLAID_ENV`. ‚úÖ

### 3. Bi-weekly reports ‚Äî TWO BUGS that need fixing

**Bug A ‚Äî cron filter queries a column that doesn't exist.**
`cron-run-reports/index.ts` does `.in("plan", ELIGIBLE_PLANS)`, but the `subscriptions` table has no `plan` column ‚Äî only `price_id`. PostgREST returns an error, `subs` is null, `userIds` is empty, and **no reports are ever generated by cron.**

**Bug B ‚Äî two competing Stripe webhook handlers writing different values.**
- `payments-webhook/index.ts` writes `price_id = session.metadata.plan` (e.g. `"auto-fill-monthly"`).
- `stripe-webhook/index.ts` (separate function) also exists and overwrites `price_id` on `customer.subscription.created/updated` with the resolved Stripe `lookup_key`.
Whichever Stripe is configured to call wins ‚Äî and the cron's `ELIGIBLE_PLANS = ["auto-fill-monthly", "finance-desk-monthly"]` only matches one of those styles. We need one webhook, one canonical value.

**Other gaps found:**
- `subscriptions.user_id` FK is `ON DELETE SET NULL` ‚Äî orphan rows after a user delete. Minor, but worth tightening to `CASCADE` so the cron can't ever attribute a stale row.
- Cron looks for `advisory_reports` table ‚Äî verify the migration exists and that `report-core.ts` writes to it.
- Instant first-report-on-checkout path lives in `payments-webhook` but not in `stripe-webhook`; if Stripe points at the latter, paying customers won't get the instant report.

### Fix plan (small, surgical)

1. **`cron-run-reports/index.ts`** ‚Äî replace `.in("plan", ‚Ä¶)` with `.in("price_id", ‚Ä¶)` and accept both naming styles (`auto-fill-monthly`, `auto_fill_monthly`, `finance-desk-monthly`, `finance_desk_monthly`) so it survives either webhook writer.
2. **Pick one Stripe webhook.** Keep `payments-webhook` (it already has the instant-first-report trigger), delete `stripe-webhook/index.ts`, and confirm `enable_stripe_payments` is pointed at `payments-webhook`. Normalize the `price_id` value written to a single canonical form (`auto-fill-monthly`, `finance-desk-monthly`, `clarity-report`).
3. **Verify the Monday 13:00 UTC pg_cron job** is actually scheduled and pointing at `cron-run-reports` with the `x-cron-secret` header from Vault. Re-run a manual trigger after the fix and confirm `candidates > 0` for a test user.
4. **Tighten `subscriptions.user_id` FK** to `ON DELETE CASCADE` via migration.
5. **Smoke-test end-to-end** in sandbox: new email ‚Üí OTP ‚Üí Google account on second device ‚Üí Plaid sandbox connect ‚Üí Stripe test checkout ‚Üí confirm instant report email arrives ‚Üí manually fire cron and confirm it skips the just-generated user (within 13-day window).
