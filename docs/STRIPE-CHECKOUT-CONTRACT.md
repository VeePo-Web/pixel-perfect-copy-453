# Stripe Checkout Contract — `create-checkout`

This is the exact contract the **Lovable-generated** `create-checkout` edge function
must satisfy so it drops into the existing frontend with no further changes. The
frontend side (`src/lib/checkout.ts`) is already built to this contract.

> Ownership: the frontend integration point, analytics, and this spec are Claude B
> (Checkout Backend & Analytics — see `COORDINATION.md`). The `create-checkout`
> edge function itself is generated by **Lovable's built-in Stripe** flow. Do not
> hand-author `supabase/functions/create-checkout/index.ts` — let Lovable write it,
> then confirm it matches this doc.

---

## How to turn it on (Lovable UI)

1. In Lovable, enable the **Stripe** integration and connect the Stripe account.
   Lovable sets the `STRIPE_SECRET_KEY` secret and generates the edge function.
2. Create a **recurring $99/mo Price** in Stripe (Product: "Auto-Fill Monthly").
   Copy its price id (`price_…`).
3. Add the secret **`STRIPE_PRICE_AUTO_FILL_MONTHLY`** = that `price_…` id.
4. (Optional) Add **`SITE_URL`** = the production origin, used as a fallback if the
   client doesn't pass return URLs.

That's it — every Auto-Fill CTA on the pricing page then opens Stripe Checkout.

---

## Request (what the frontend sends)

`supabase.functions.invoke("create-checkout", { body })` with:

```jsonc
{
  "plan": "auto-fill-monthly",                       // CheckoutPlan
  "successUrl": "https://site/?checkout=success#/pricing",
  "cancelUrl":  "https://site/?checkout=cancel#/pricing#auto-fill"
}
```

The `?checkout=` query is intentional — the app is hash-routed, so the query rides
**before** the `#` fragment and is read back via `window.location.search` on return
by `handleCheckoutReturn()`.

## Response (what the function must return)

```jsonc
{ "url": "https://checkout.stripe.com/c/pay/cs_test_…" }
```

The frontend redirects to `url`. Anything else (error, missing `url`) makes the CTA
fall back to the offer section — never a dead click.

---

## What the function must do

1. Handle the CORS preflight (`OPTIONS`).
2. Read `plan`, `successUrl`, `cancelUrl` from the JSON body.
3. Map the plan → Stripe Price. **Do not silently default** — an unknown plan must
   throw, not quietly map to the wrong price:
   - `"auto-fill-monthly"` → `Deno.env.get("STRIPE_PRICE_AUTO_FILL_MONTHLY")`
4. Create a Checkout Session:
   - `mode: "subscription"`
   - `line_items: [{ price, quantity: 1 }]`
   - `success_url: body.successUrl ?? \`${SITE_URL}/?checkout=success#/pricing\``
   - `cancel_url:  body.cancelUrl  ?? \`${SITE_URL}/?checkout=cancel#/pricing#auto-fill\``
   - `allow_promotion_codes: true` (optional)
5. Return `{ url: session.url }`.
6. If `STRIPE_SECRET_KEY` is missing, return a clear error (the frontend already
   degrades gracefully) — never 200 with no `url`.

### Secrets
| Secret | Set by | Purpose |
|---|---|---|
| `STRIPE_SECRET_KEY` | Lovable (on connect) | Stripe API auth |
| `STRIPE_PRICE_AUTO_FILL_MONTHLY` | You (paste `price_…`) | The $99/mo recurring price |
| `SITE_URL` | You (optional) | Return-URL fallback if client omits them |

## Source of truth — the webhook (recommended next, Claude B)

The success URL is **not proof of payment** — a buyer can reach it without a
completed charge, and Stripe can complete a subscription even if the browser never
returns. So:

- `analytics.purchaseCompleted` (fired from `handleCheckoutReturn`) is an
  **optimistic, marketing-only** signal. Fine for ad pixels; not for billing.
- The **billing source of truth** is a `stripe-webhook` edge function handling
  `checkout.session.completed` / `invoice.paid` (and the refund/`customer.subscription.deleted`
  counterparts), verifying the Stripe signature and recording the subscription in
  a `subscriptions` table. This is the recommended follow-up and is in Claude B's
  lane — it does not block turning Checkout on for a done-for-you service where the
  payment itself is the deliverable trigger.

## What's still needed for a visible confirmation (Claude A — conversion)

`handleCheckoutReturn()` returns `"success" | "cancel" | null` for a UI surface to
show a confirmation/thank-you state on the pricing page. Building that **screen** is
the conversion build's call — this module only detects the return and fires the
event.
